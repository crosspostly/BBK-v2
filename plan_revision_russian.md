Спасибо за уточнение! Это очень важная информация, которая значительно меняет подход к работе с расписаниями.

Я понял, что каждое заведение будет иметь свой отдельный лист (вкладку) с расписанием в Google Таблицах. Администратор будет предоставлять ссылку на эти листы, и бэкенд должен будет собирать информацию со всех таких листов в "общий календарь".

Чтобы это реализовать наиболее эффективно и масштабируемо, я предлагаю следующий подход:

**1. Централизованный список заведений:**
   Вместо того чтобы вручную указывать каждый лист с расписанием, мы создадим **отдельную Google Таблицу (или отдельный лист в существующей таблице 'Панда Ролс') под названием "Заведения" (Establishments)**.
   В этой таблице будут храниться следующие данные для каждого заведения:
   *   **Название заведения:** (Например, "Панда Ролс")
   *   **ID/URL Google Таблицы с расписанием:** Ссылка или ID конкретной таблицы/листа, где находится расписание данного заведения.
   *   **Название листа с расписанием:** (Например, "Расписание")
   *   **Категория:** (Опционально, для будущей фильтрации, например, "Рестораны", "Салоны красоты" и т.д.)

   Таким образом, когда администратор добавляет новое заведение, он просто добавляет новую строку в эту центральную таблицу "Заведения".

**2. Обновление бэкенда (файл `main.py`):**
   *   **Динамическое получение расписаний:** Бэкенд будет сначала читать список заведений из таблицы "Заведения". Затем для каждого заведения он будет динамически подключаться к его отдельному листу с расписанием, получать данные и объединять их в единый "общий календарь".
   *   **Структура данных:** Мне нужно будет немного изменить структуру данных, которые возвращает `GET /api/schedule`, чтобы она могла включать информацию о заведении для каждого слота (например, `[{establishment_name: "Панда Ролс", date: "...", slot: "..."}, {establishment_name: "Кафе Солнце", date: "...", slot: "..."}, ...]`).
   *   **Бронирование:** При бронировании слота, Mini App будет отправлять на бэкенд не только дату и номер слота, но и **ID или название заведения**, чтобы бэкенд знал, в какой конкретный лист нужно вносить запись.
   *   **Мои записи:** Для просмотра своих записей пользователем, бэкенд будет сканировать все листы расписаний всех заведений, чтобы найти записи данного пользователя.

**Преимущества такого подхода:**
*   **Гибкость:** Легко добавлять/удалять новые заведения без изменения кода.
*   **Масштабируемость:** Поддерживает множество заведений.
*   **Удобство для админа:** Управление списком заведений через Google Таблицу.
